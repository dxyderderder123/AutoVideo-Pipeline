## ⚙️ 项目架构与工作流程

本项目实现了从**输入文章到视频成品**的全链路自动化逻辑。系统具备硬件感知（GPU锁）、API熔断降级和多线程并行能力，确保了生产环境的稳健性。

### 1. 核心自动化管线 (Pipeline)

系统将视频制作逻辑拆解为 9 个独立步骤，并由核心逻辑统筹调度：

| 阶段 | 功能描述 | 核心技术 |
| :--- | :--- | :--- |
| **01 分析** | 语义切片与关键词提取 | LLM (NLP 语境分析) |
| **02 配音** | 高拟真语音合成 | **VibeVoice** 推理引擎 |
| **03 素材** | 视觉素材抓取与熔断保护 | Pexels/Pixabay API 接入 |
| **04 对齐** | 音频与文本毫秒级强制对齐 | WhisperX / 词频比例算法 |
| **05 字幕** | 视觉短句边界切分 | 动态文本折行逻辑 |
| **06 翻译** | 双语字幕生成与渲染 | LLM 翻译 + libass 渲染 |
| **07 混剪** | 视音频底层物理合成 | **FFmpeg** 媒体处理滤镜链 |
| **08 封面** | AI 艺术封面生成 | 扩散模型 (Diffusion Model) |
| **09 发布** | 自动化分发与上传 | 协议指纹模拟 (WAF 对抗) |

### 2. 批量生产能力

本项目支持**一键扫描式批量生产**：

- **自动化入口**：系统自动扫描 `workspace/input/` 目录下的所有 Markdown 文稿。
- **并行调度**：自动检测 GPU 显存负载，动态决定并行任务数，在保证硬件安全的前提下最大化生产效率。
- **一键运行**：通过 `scripts/run_batch.bat` 即可启动完整的流水线任务。

## 📈 开发动机与技术历程 (Motivation & Evolution)

本项目的产出不是一次简单的代码编写，而是作者在过去一年中，在平衡考研备考、探索经济独立以及追求物理与 AI 交叉研究过程中的一个阶段性记录。

### 1. 缘起：备考压力下的自动化探索
在准备物理考研初试期间，由于希望在不依赖家庭经济支持的前提下高效复习，我开始尝试通过 AI 自动化流程运营科普自媒体（哲学科普账号“哲明星”，现关注量 5.5 万）。
*   **初期瓶颈**：一年前，受限于 AI 对精确数学/几何关系的生成能力，我将方向从物理科普转向更具意象表达空间的哲学科普。
*   **半自动化阶段**：当时采用 ComfyUI（生图）+ GPT-SoVITS（声音克隆）的架构。流程涉及“原文提取 -> Gemini 脚本生成 -> Python 素材生成 -> 人声降噪/混响处理 -> PR 辅助剪辑”。
*   **存证**：此阶段的原始脚本和素材处理文件，已归档至项目根目录的 `archive/old_workflow/` 文件夹中。

### 2. 共鸣：从“应用探索”向“AI for Science”的转向
在备考过程中，我关注到王磊老师在“物理与人工智能交叉领域”的研究方向，这让我意识到过去两年的 AI 编程探索（AI Coding）不应仅仅停留在应用层的自媒体工具，而应转向更为严谨的学术研究（如 AI 驱动的物理模拟与自动化验证）。这成为我报考物理所、深耕 AI4S 方向的核心动力。

### 3. 攻坚：关于 Remotion 与 Manim 全自动化的试错
初试结束后，我曾计划利用一个月时间通过 **Playwright**（自动化控制）+ **Remotion**（前端视频框架）+ **Manim**（物理动画引擎）构建一套具备理科展现力的智能 Agent：
*   **进展**：成功实现了基于 Playwright 的文稿与脚本一键生成自动化。
*   **挫败**：但在渲染阶段遇到了巨大瓶颈——Remotion 产出的视频在全量自动化下视觉质量不稳（类似动态 PPT），而 Manim 在自动化脚本驱动下的几何逻辑表现极度不稳定。
*   **反思**：这段失败的经历让我深刻理解到高层框架在工业级自动化一致性上的局限，但也锻炼了我对异构工具链（Playwright/Manim）的深度调试能力。

### 4. 现状：务实选择下的工程衍生品
在临近复试、全力投入《普通物理》与《四大力学》复习的有限时间内，我果断止损，决定从之前的复杂失败中提取出最稳健的技术组件，利用 **Python + FFmpeg** 原生重构了目前这套英语视频工作流。
*   **定位**：本项目是作者工程经验的一个坚固“衍生品”，它舍弃了不稳定的上层框架，优先保证 100% 的生产闭环与系统稳健性。
*   **价值**：它证明了作者能够在资源限制（硬件负载、API 成本）与时间压力（专业课复试准备）下，迅速交付高质量工程方案的执行力。
